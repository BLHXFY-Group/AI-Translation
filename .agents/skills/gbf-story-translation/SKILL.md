# GBF 剧情翻译

## 核心规则

* 将任务视为日文游戏文本初译或译文审校，而不是编程任务。
* 以当前行的日文 `text` 和相邻日文剧情为事实依据；当前行 `text` 是该行译文的直接依据。审校时，既有 `trans` 只用于发现问题和比较改动，与日文冲突时不得作为事实来源。
* 只修改用户指定范围内的 `trans` 列；`name` 列不属于翻译、审校或修正范围，必须完整保持原样。
* 可以读取 `name` 以识别说话者，但不得把其中已有的中文显示名当作 `trans` 的人名译法依据，也不得让各文件的 `trans` 分别跟随各自的 `name`。
* 保留行顺序、ID、日文原文、标签、CSV 结构、文件编码、BOM 状态和 CSV 记录分隔符风格（CRLF 或 LF）。
* 目标 CSV 的每条记录必须占且仅占一个物理行；任何字段内部都不得包含真实的回车或换行字符。
* 默认不要修改 `译者` metadata；除非用户明确要求，否则完全保持原样。
* 纯文本翻译工作不要运行游戏、构建、导入、导出或任何运行时命令；只做静态检查。
* 活动剧情按批次处理时，将批次开始校准视为强制前置检查点；第一批、后续批次以及中断后恢复处理时，都必须先快速回顾“翻译风格”和“断行与标记”，完成后才能开始初译或审校本批正文。
* 不要添加日文原文或紧邻上下文中不存在的设定解释、情绪、因果关系或剧情信息。

## 工作模式

1. 用户要求翻译且目标 `trans` 为空时，使用初译模式；逐条阅读日文原文和相邻上下文后填写 `trans`。待翻译记录必须在用户指定范围内，且 `text` 含有应写入 `trans` 的可见内容，包括有意义的纯标点台词；`text` 为空、metadata、无可见内容的纯控制或纯标签记录，以及用户明确要求留空的记录，不属于待翻译记录。
2. 用户要求检查、审校、润色、修正或重译已有内容时，使用审校模式。
3. 同一目标范围的 `trans` 既有空白又有已译内容时，用户只说“翻译这个文件”或同等泛称时，默认只填写待翻译记录的空白 `trans`，已有译文保持不变；用户明确要求“翻译并检查”、审校或重译时，才同时处理已有译文。
4. 初译模式下不覆盖目标范围内已有的 `trans`，除非用户明确要求同时审校或重译。
5. 初译时不得仅根据其他行或其他文件的中文 `trans` 生成当前译文；已稳定的项目译法只能用于人名、术语和文风一致性，当前行含义仍以当前日文 `text` 为准。

## 剧情顺序

处理角色剧情文件夹时：

1. 将文件名包含 `fate` 的文件视为角色第一篇剧情。
2. 其余剧情文件按数字 `q` 编号，再按数字 `s` 编号处理。
3. 使用数值顺序，不使用字典序；例如 `q2` 应排在 `q10` 之前。
4. 按这个顺序维持术语、人物塑造和剧情指代的一致性。

处理活动剧情文件夹时：

1. 将文件名以 `scene_evt` 开头的 CSV 视为活动剧情文件。
2. 示例中的活动 ID，如 `260529`，只是占位示例；实际活动 ID 会随活动变化。
3. 将类似 `scene_evt260529_cp0_q1_s10.csv` 的文件视为序章；分批处理时并入第一节所在批次，不要作为独立首批处理。
4. 将类似 `scene_evt260529_cp0_q2_s10.csv` 的文件视为终章；分批处理时并入最后一节所在批次，不要作为独立首批处理。
5. 叙事顺序为序章、其余活动剧情文件、终章；其余活动剧情文件按数字 `cp`、数字 `q`、数字 `s` 的顺序处理。
6. 使用数值顺序，不使用字典序；例如 `cp2` 在 `cp10` 之前，`q2` 在 `q10` 之前，`s2` 在 `s10` 之前。
7. 按这个顺序维持活动术语、伏笔、人物塑造和剧情指代的一致性。

活动剧情处理粒度：

1. 处理活动剧情目录时，先只查看文件名、文件数量和大小来判断规模；不要一开始全量读取所有 CSV 内容。
2. 文件较少时，可以按 `cp/q/s` 数值顺序连续处理。
3. 活动剧情目录中 `scene_evt*.csv` 超过 8 个，或总大小超过约 150 KB 时，视为较大目录，默认按 `cp` 分批处理；每批只读取当前 `cp` 需要初译或审校的 CSV。
4. 单个 `cp` 超过 6 个文件，或总大小超过约 90 KB 时，再按数字 `q` 分批；每个批次内部继续按 `s` 数值顺序处理。
5. 用户明确要求一次处理全部时，可以在同一任务中依次完成所有批次，但仍须按下述规则分批读取、校准、初译或审校；开始前说明批次安排及跨批次一致性风险。
6. 分批时不要把 `cp0` 单独作为一个批次；第一批应包含序章和第一节，最后一批应包含最后一节和终章。
7. 每完成一个活动剧情批次，整理本批出现的角色称呼、术语、地名、组织名、关键剧情概念和特殊语气；下一批开始前沿用这些临时记录，避免同一活动内译法漂移。
8. 每个批次开始前都必须完成批次校准，第一批也不例外；中断后恢复处理时重新校准。第一批快速回顾“翻译风格”和“断行与标记”；后续批次还要先回顾上一批临时记录。未完成校准，不得开始初译或审校本批正文。
9. 初译批次必须按 `未校准 → 已校准 → 逐行翻译中 → 已逐行翻译 → 已验证` 推进；审校批次必须按 `未校准 → 已校准 → 逐行审校中 → 已逐行审校 → 已验证` 推进。混合模式使用 `逐行处理中 → 已逐行处理`。不得跳级；未读完本批全部目标记录不得标记为已逐行处理；未完成静态检查不得标记为“已验证”；当前批次未达到“已验证”时，不得开始下一批或宣称该批完成。
10. 对较大活动目录，实际初译或审校必须按批次推进；不得用全目录脚本替代逐批阅读、校准和人工翻译或审校。脚本只能用于统计、候选问题扫描、结构校验，或经逐条确认后的定点写回。脚本扫描、术语或引号批量替换、摘要检查、格式修复和结构验证都不计入逐行正文处理，也不能作为“已逐行翻译”、“已逐行审校”或批次完成的依据。
11. 不要把跨批次临时记录写进 CSV 或新增说明文件，除非用户明确要求；它只作为当前翻译或审校过程中的工作上下文。

## 翻译风格

活动剧情的批次开始校准不需要完整重读本 Skill，只需快速确认翻译依据、跨批次术语和角色口吻延续、`trans` 人名不跟随各文件 `name` 漂移、准确性优先级、情绪强度与台词自然度；同时按“断行与标记”确认断行和引号规则。

* 优先遵循日文原文语气，再改写成自然的简体中文 UI 文本。
* 原文语气允许时，保持轻快的《碧蓝幻想》冒险氛围：叙述清楚、对话活泼、幽默克制；严肃、悲伤或紧张场景以原文氛围为准。
* 先根据日文原文和周边剧情判断角色口吻，再撰写或重写中文；不同角色的说话方式要有区分。
* 保持信息完整。初译不得漏掉目标正文行；审校时修正明显误译，尤其注意施事者、代词、关系逻辑、责任、义务，以及“谁对谁做了什么”。
* `trans` 文本中的人名、敬称、组织、船名、地名、能力名和反复出现的剧情概念，要与同一文件夹、同一角色系列和附近剧情文件保持一致。
* 确定 `trans` 中的人名译法时，优先采用用户明确指定的译法，其次采用同一角色系列或同一剧情范围内已经稳定的项目译法；没有稳定译法时，再根据日文和上下文确定，并在本次任务范围内统一使用。
* `name` 中的中文显示名不属于上述译法依据。若它与 `trans` 应采用的统一人名译法不一致，保持 `name` 原样，并在 `trans` 中使用统一译法。
* 如果本地上下文中没有明确既定译法，根据日文含义翻译，并在本次任务范围内保持一致。
* 不要过度直译反复出现的剧情专有概念；根据日文原意和上下文区分相近术语。
* 翻译或审校单条 `trans` 时，优先级如下：
  1. 保留事实含义和说话者意图。
  2. 保留角色口吻和人物关系动态。
  3. 写成适合游戏 UI 的自然简体中文。
  4. 只在前三项成立的前提下提升措辞和流畅度。
* 在准确保留日文含义、说话者意图和角色口吻的前提下，调整语序、删去生硬直译痕迹、补足必要的轻微衔接，让译文更顺、更凝练、更有台词感；不要增加原文没有的信息、情绪或修辞，也不要改写剧情逻辑或人物态度。
* 叙述应清楚、有画面感但不过度文学化；对白应自然、有角色感，不过度口语化、网络化或夸张化。
* 当“更好听”和“更准确”冲突时，优先准确；当“更华丽”和“更像原角色”冲突时，优先角色口吻。
* 保留原文的情绪强度，不把普通叙述、拌嘴或轻微情绪拔高成更戏剧化、文学化、搞笑、讽刺、可爱、幼稚或网络化的表达；中文语气词和感叹号只在原文语气支持或自然表达确有需要时使用。

## 断行与标记

* 不要机械照搬日文 `\n`。
* 严格区分“显示换行”和“物理换行”：
  * 游戏 UI 的显示换行必须在字段内写成字面量 `\n`，即反斜杠和字母 `n` 两个字符；它不是 CSV 物理换行。
  * 每条 CSV 记录（包括表头和 metadata）必须完整保存在一个物理行内；字段内部禁止真实的 `CRLF`、`LF` 或 `CR`。
  * 真实的 `CRLF` 或 `LF` 只用于分隔 CSV 记录，并按原文件的记录分隔符风格保存。
* 对普通对话和叙述的 `trans`，按中文语义、台词节奏和 UI 阅读自然度调整 `\n`：
  * 最多 3 条显示行。
  * 能自然单行呈现的短句、短引语和完整语义块，不要强行拆行。
  * 不要拆开固定词组、专有名词、修饰关系、省略号、标签内容或完整语义块。
  * 断行位置优先选择自然语义停顿处，如句号、逗号、转折、并列、原因/结果、称呼后停顿等。
* 断行位置必须逐条人工判断。脚本可以检测候选问题；人工确认每条译文后，可以按 `文件 + id` 定点写回，但不得用长度阈值或其他算法自动决定断行位置并批量改写。
* `0-synopsis` 是摘要字段。除非原字段格式或用户明确要求，否则保持为一个自然段。
* 除非文件格式明确要求显示断行，否则不要把 `0-synopsis` 纳入普通对话断行检查。
* 保留原有标记标签，如 `<span class='scene-font-4'>...</span>`，只翻译其中可见文本。
* 原文使用 `「」` 时，译文保留日式引号 `「」`；不要替换成中文弯引号 `“”`。
* 原文使用书名号式引号如 `『』` 时，在中文中保留合适的日式引号形式

## CSV 编辑流程

1. 编辑前先检测原文件编码、BOM 状态和 CSV 记录分隔符风格。
2. 编辑前检查目标文件是否已经存在字段内真实换行或 CSV 解析错误。若异常位于本次可编辑的 `trans`，按本节规则修正；若位于 `name`，只报告并保持原样；若位于 `id`、日文 `text` 或 metadata，只报告并请求用户授权，不得为通过检查而静默修改。
3. 确认为 UTF-8 时按 UTF-8 读取；如果原文件是 UTF-8 无 BOM，保存时也保持 UTF-8 无 BOM。
4. 尽可能使用 CSV 解析器和列名处理；不要用简单逗号分割来编辑 CSV。
5. 只修改少量 `trans` 单元格时，优先使用 `apply_patch`。
6. 如果需要脚本化改写，范围必须严格限定在目标 `trans` 单元格，并保留编码、BOM 状态、行顺序、CSV 记录分隔符风格和 CSV 引号。
7. 修改多行或多文件时，优先使用 CSV 解析器配合 `文件 + id -> new_trans` 定点更新表写回；需要防误改时，再记录 `文件 + id -> expected_old_trans` 并在写入前校验旧值。不得因为使用脚本而省略对每条新译文或修订译文的人工确认。
8. 脚本化写回 `trans` 前，必须对待写入的字段值做防御式规范化：
   * 先把字段值内部真实的 `CRLF` 转成字面量 `\n`，再把剩余的单独 `LF` 和 `CR` 转成字面量 `\n`；转换顺序不得颠倒，以免一个 `CRLF` 变成两个显示换行。
   * 这里规范化的是“字段值”，不是序列化后的整个 CSV 文件；不得替换用于分隔记录的真实换行。
   * 最终字段值中保存的是反斜杠和字母 `n` 两个字符；在 Python 字符串字面量中通常写作 `'\\n'`。
   * 不要让 Python、编辑器或补丁中的多行字符串直接落入 CSV 字段。
9. 不要把全文 `replace(old, new)` 作为常规改写方式。只有极少量、机械性、限定在目标文件内且确认不会误伤重复文本的修正，才可以使用字符串替换；替换前必须列出候选命中，替换脚本必须检查命中次数，避免静默失效。
10. 脚本化写回后仍要重新解析 CSV，并对比 `id`、`name`、日文 `text`、记录数和记录顺序是否保持不变。

## 编辑后的静态检查

编辑后分两层做静态检查。

对整个目标 CSV 做结构与不变量检查：

* CSV 仍能正常解析，引号、逗号和记录分隔符仍然有效。
* 对比编辑前后的解析结果：记录数和记录顺序不变；每个 `id`、`name` 和日文 `text` 字段完全不变；只有目标 `trans` 单元格发生改动。
* 除非用户明确要求修改，`译者` metadata 保持不变。
* 每条 CSV 记录仍只占一个物理行，任何字段都不包含真实 `CR` 或 `LF`，显示换行只使用字面量 `\n`；物理行数等于 CSV 解析后的记录数（含表头）。

只对本次用户指定的目标 `trans` 做翻译质量检查：

* 初译模式下，确认待翻译记录没有意外留下空白 `trans`，并确认原本已有的 `trans` 未被覆盖。
* 没有残留中文弯引号 `“` 或 `”`。
* `trans` 中没有明显未翻译的日文残留；姓名、术语、标签或有意保留的原文除外。
* `trans` 中的术语和人名与本地术语及上下文一致；同一人物没有按不同文件的 `name` 机械切换译法。
* 普通正文 `trans` 不超过 3 条显示行。
* 对明显过长的行进行复查；判断 UI 可见文本长度时忽略标记标签本身的长度。
* 标记标签结构保持完整。
* 非目标 `trans` 中发现的既有翻译质量或断行问题只报告，不得为了通过本次检查而修改。
* 活动剧情如果按 `cp` 或 `q` 分批处理，全部批次完成后再对全部已完成批次的目标 `trans` 做一次跨批次质量复查：重点检查人名、术语、地名、组织名、关键剧情概念和中文弯引号；同时对全部目标 CSV 重做可解析性和不变量检查。

## 常见陷阱

* 不要把“翻译空白 `trans`”自动扩大为审校或覆盖已有译文。
* 不要因为其他文件有相似中文句子，就跳过对当前日文 `text` 的逐条理解。
* 不要在既有中文译文与日文冲突时，从既有中文译文反推含义。
* 不要脱离日文原文，只依据既有 `trans` 做二次润色；可以保留准确自然的现有表达，但每项实质改动都必须由日文原文和上下文支持。
* 短中文引语可以自然单行呈现时，不要在其中添加不必要的 `\n`。
* 不要把游戏显示用的字面量 `\n` 写成 CSV 字段内真实换行；这种错误仍是合法 CSV，普通解析检查不会自动发现。
* 不要只为缩短文本而删掉自然的中文时态、体貌或完成标记。如果日文表达了完成或获得状态，且中文用 `了` 更准确流畅，就应保留。
* 不要把沙箱提权失败误判为项目文件夹权限问题；在本工作区优先使用 `apply_patch`。
* 不要仅因其他译法在别处更常见，就悄悄把术语从本地项目用法改走。

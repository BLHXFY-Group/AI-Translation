# 翻译与审校规则

## 翻译依据

* 以当前日文 `text` 和必要的日文上下文为事实依据。既有 `trans` 只供审校；冲突时以日文为准。
* 允许范围内的其他中文译文只能用于稳定人名、术语和文风，不能代替当前日文或成为剧情事实来源。
* 不添加原文不存在的设定、情绪、因果、修辞或剧情信息。

## 准确性与表达

* 优先保证事实、说话者意图和角色关系，再追求自然简体中文和润色。
* 逐条核实施事者、代词、人物关系、时态、完成状态和“谁对谁做了什么”。
* 可以调整语序和轻微衔接，但不得改变剧情逻辑、人物态度或原文情绪强度。
* 避免生硬直译，也不要擅自文学化、网络化或强化搞笑、讽刺、可爱等风格。

## 角色与术语一致性

* 根据日文和周边剧情区分角色口吻；人名、敬称、组织、地名、能力名和关键概念在连续任务内保持一致。
* 译法优先采用用户指定，其次采用当前 `terms.md` 已确定的译法，再参考允许搜索范围内的稳定既有译文。
* 核对既有译名时，只能在当前目标 CSV 的直接父目录内非递归搜索，并且仍只可读取用户已提供且已进入当前处理批次的文件。禁止搜索父目录、其他目录、仓库其余位置或外部来源；允许范围内没有可靠依据时，直接依据当前日文和上下文确定，不得扩大范围。
* `name` 可用于识别说话者，但其中的中文显示名不是人名译法依据；不一致时保持 `name` 原样，在 `trans` 使用统一译法。
* 每个文件完成后只记录后续仍可能使用且容易漂移的用户指定译法、任务特有或歧义专名、非显然称呼、角色口吻和未结束状态。
* 不因词语首次出现就自动记录；不记录译法无歧义且可从当前正文直接判断的人名和基础世界观词、普通词典词、已结束剧情、处理历史或文件顺序。不得为判断是否需要记录而扩大搜索范围，并及时删除不再具有前向作用的条目。

## 断行与标记

* 按中文语义、台词节奏和 UI 阅读自然度决定换行，不机械照搬日文；普通正文最多 3 条显示行。
* 不拆开固定词组、专名、修饰关系、标签或完整语义块；优先在自然停顿处断行。
* 显示换行使用字面量 `\n`，不得写入真实换行；断行由 Agent 判断，不按长度自动批量改写。
* 保留标签及属性，只翻译可见文本。原文已有引号时，保持层级、数量、顺序和类型：`「」` 对应 `「」`，`『』` 对应 `『』`，`“”` 对应 `“”`；不得在三种引号之间互换。

## 特殊记录

* `0-chapter_name`、`0-synopsis` 是有效正文；`0-synopsis` 通常保持一个自然段。
* `译者` 的 `trans` 只能写入已知贡献信息，不得编造译者或审校者；不适用角色口吻、三行和日文残留规则。
* 纯控制记录不要无理由填充；有意义的纯标点台词必须结合上下文处理。

## 人工处理边界

* 用户指定的文件、ID 和保持不变范围必须由 Agent 遵守，脚本不会替代该判断。
* 每条目标正文都要阅读日文、必要上下文和已有译文，并人工确认修改或保持。
* 首次理解时同时完成翻译或审校、口吻、断行和标签自检；写回后不再全面逐条通读。
* 摘要、搜索、格式检查和验证脚本都不能替代逐条翻译审校。
* 只生成更新 JSON 并调用通用脚本写回；脚本失败时修正输入，不得绕过脚本直接修改 CSV、state 或原始副本。

## 质量复查

* 写回后只处理脚本错误与警告、术语候选和 backlog，不重新全面通读。
* 后续文件为前文提供新依据时先登记 backlog，不立即修改已验证旧文件。
* 最终只处理脚本错误与警告和 backlog，不另行执行全量术语、称呼或定点一致性搜索。
